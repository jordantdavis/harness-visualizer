# Claude Code Harness Visualizer — Roadmap

A local-only, single-user developer tool for **seeing what your Claude Code harness
actually does** — which hooks fire, in what order, with what data, and how long tools
take. Config is intent; the captured event stream is reality. This is a reality-viewer.

Written in Go. Synthesized from architecture, product, and UX review (2026-05-28).

---

## Guiding principles

- **Single-user, local, personal.** No auth, no TLS, no multi-tenancy, no cloud.
  Binding to `127.0.0.1` only **is** the entire security model.
- **Spend design budget on CLI critical-path safety.** The hook CLI runs inside every
  tool call; it must never slow down or break Claude Code. Everything else stays boring.
- **A Go channel is the queue. JSONL files are the database.** No broker, no SQLite.
- **Capture is the contract.** If the tool is ever lossy or misorders events, it loses
  trust and gets abandoned. The daemon owns authoritative ordering.
- **YAGNI ruthlessly.** This domain is fascinating and every adjacent feature feels like
  "insight." If it doesn't help answer *"what just fired, with what data?"*, it waits.

## Architecture

### One binary, three roles

`cchv` with subcommands (simplest install — the plugin manifest references one path):

- `cchv hook` — hook forwarder (also the bare/default invocation)
- `cchv daemon` — long-running HTTP server (`--foreground` for debugging)
- `cchv tui` — bubbletea UI

### Package layout

```
cmd/cchv/main.go            # subcommand dispatch
internal/event/             # canonical Event type — the shared contract
internal/client/            # hook-side: read stdin, enrich, POST, fail-safe
internal/daemon/            # HTTP server + in-memory hub (SSE fan-out)
internal/store/             # per-session JSONL writer + readers
internal/paths/             # XDG dirs, port file, pidfile
internal/tui/               # bubbletea models/views
plugin/                     # Claude Code plugin manifest (static hooks config)
```

### Data flow

- **Live:** hook → `cchv hook` (stdin) → `POST /events` → daemon returns `204`
  immediately after accepting bytes → fans out to (a) per-session JSONL writer,
  (b) in-memory hub → **SSE** stream to TUI.
- **Historical:** TUI → `GET /sessions` (list) and `GET /sessions/{id}/events` (read).
- **The TUI reads everything through the daemon** (SSE live, HTTP history) — it never
  tails JSONL files directly. One read path, one source of truth.

### HTTP API (5 routes, no versioning)

| Route | Purpose |
|---|---|
| `POST /events` | CLI → daemon; one event; returns `204` fast |
| `GET /sessions` | list: id, start time, event count, cwd, last activity |
| `GET /sessions/{id}/events` | historical read (supports `?since=seq`) |
| `GET /stream` | SSE live fan-out (optional `?session={id}`) |
| `GET /healthz` | liveness; used by CLI auto-spawn check |

### Event schema — canonical envelope + raw passthrough

```go
type Event struct {
    ID         string          // generated by CLI (uuid/ulid)
    CapturedAt time.Time       // CLI wall-clock at capture
    Seq        int64           // daemon-assigned, per-session monotonic (ordering/cursor)
    HookEvent  string          // from hook_event_name
    SessionID  string
    CWD        string          // omitempty
    ToolName   string          // promoted for TUI columns; omitempty
    Raw        json.RawMessage // the ENTIRE original hook payload, verbatim
}
```

- `Raw` is the escape hatch: all hook-type-specific fields (`tool_input`,
  `tool_response`, etc.) live there untouched. We never break on a new field and never
  need migrations.
- Promote only what the TUI renders as columns/filters (`HookEvent`, `ToolName`, times).
  Parse defensively in the CLI — if a field is absent, leave it empty, never fail.
- **`Seq` is daemon-assigned** (authoritative ordering, immune to delivery jitter);
  `CapturedAt` is for display only.
- On disk: one marshaled `Event` per JSONL line. The file IS the log.

### Daemon lifecycle & storage

- **Auto-spawned by the CLI** on connection-refused: fork `cchv daemon` fully detached
  (`Setsid`, logs to file), do **not** wait, drop the current event, exit 0. Next hook
  (ms later) hits the running daemon. Manual `cchv daemon` is the override / dev path.
- **Single instance** via port bind (the bind is the mutex) + pidfile. **Default port
  7842; on conflict fall back to an ephemeral port and write the actual port to a port
  file** that the CLI and TUI read.
- **Storage:** `$XDG_DATA_HOME/cchv/sessions/{session_id}.jsonl` (with OS fallbacks).
  Runtime files (port, pid, daemon log) under `$XDG_RUNTIME_DIR/cchv/` or data-dir fallback.
- **Concurrency:** daemon is the only writer. `POST` handler → buffered channel →
  one writer goroutine per session. Buffered writes; periodic flush (~250ms) and flush
  on `Stop`/`SessionEnd`. Close idle session file handles after N minutes. No file locking.

### CLI critical-path safety (load-bearing)

**Hooks are async / non-blocking.** Claude Code must never wait on the visualizer. The
`cchv hook` process does the minimum synchronously, returns immediately, and never gates
the agent on forwarding completion. The budget below is a hard ceiling, not a target —
the goal is "imperceptible," and the daemon (not the hook) is the buffer.

- ~100ms total budget (`context.WithTimeout`); POST timeout ≤50ms.
- **Always exit 0.** Emit nothing on stdout (pure observer). Silent stderr unless a
  debug env var is set.
- Fire-and-forget; swallow all errors (refused/timeout/5xx).
- Bounded stdin read (`io.LimitReader`, ~10MB cap).
- Top-level `defer recover()` — a bug must never surface as a hook failure.
- Dial `127.0.0.1:PORT` directly (no DNS, avoid `localhost` resolver edge cases).
- Keep the hook-path build tiny: `internal/client` must NOT import `internal/tui`.

### TUI UX

- **Master-detail** (`lazygit`/`k9s` model): `Sessions │ Events │ Inspector`.
  Responsive collapse: ≥120 cols = 3-column; 80–119 = 2-pane + inspector drawer;
  <80 = single-pane stacked with breadcrumb. Top status bar + bottom context-sensitive
  keyhint bar.
- **Event row:** `time · status-glyph · hook · tool · target-gist · duration` with
  fixed-width columns so the eye scans down. Status = **glyph + color (redundant)** so
  it survives `NO_COLOR`, SSH, and color-blindness. Right-aligned duration column.
- **Live = `tail -f` mental model.** "Mode" is emergent from selection + tail position,
  not a hard toggle. Follow auto-pauses on manual scroll; buffers new events with an
  `↓ N new` count; `G` jumps to live and re-follows. Heartbeat/sparkline shows
  streaming vs `idle Ns`.
- **Folded Pre/Post operations** are the default (one row flips `▶`→`✔`/`✘` with
  duration); `o` toggles to flat chronological.
- **Filter grammar** (`/`): `tool:Bash hook:PostToolUse status:error dur:>500ms path:auth`;
  bare text = fuzzy match. Filters persist across live/historical. Error-hop (`e`) +
  sticky error tape.
- **Inspector:** foldable JSON tree with a fixed key-facts header; syntax-aware value
  rendering (commands, diffs, paths); `y`/`Y` yank; `r` raw escape hatch.
- **Resilience:** malformed event → degraded row, never crash. `--plain` line-per-event
  mode for screen readers / bug reports / piping.

### Color & accessibility

- Color is always **redundant** with glyph + position + label. Target the 16-color ANSI
  palette (respects user theme), opt up to 256/truecolor if advertised. Use lipgloss
  `AdaptiveColor` for light/dark. `NO_COLOR` / dumb-term render uses glyphs + bold +
  reverse-video and text tags (`[OK]`/`[ERR]`/`[RUN]`) — designed first, not last.
- Color-blind safe: success `✔` vs error `✘` differ in shape; never red/green adjacency
  as the only difference. Honor `--no-animation` / reduced motion (no blinking).

### Keymap (vim-style; depth × pane model)

`Tab`/`1 2 3` move between panes · `j/k` `Ctrl-d/u` `g/G` motion · `Enter`/`l` deeper,
`Esc`/`h` back · `f` follow · `space` pause · `/` filter · `n/N` matches · `e` next error ·
`o` fold toggle · `y`/`Y` yank · `r` raw · `?` help overlay · `q` quit.

## Cut hard (YAGNI — revisit only on real signal)

- SQLite / any DB / indexes — JSONL is greppable; `ripgrep` covers cross-session search.
- WebSockets — SSE is one-way and less code.
- Auth / TLS / API tokens / CORS — localhost only.
- Message queue / broker / protobuf / gRPC — a Go channel + JSON over HTTP.
- Config files, schema versioning / migrations — env-var overrides; `Raw` passthrough.
- CLI local spool/WAL — add only if cold-start event loss proves annoying.
- Cross-session analytics, replay/re-injection, export, web UI, alerting.
- Search beyond in-session filter; payload-schema diffing across CC versions.

---

## Success criteria (behavioral, not metrics)

- **Truth test:** it shows you something about your harness you didn't know / were wrong about.
- **Reach test:** when you change a hook, your reflex becomes "check the visualizer"
  instead of `echo >> /tmp/debug.log`.
- **Trust test:** you believe what it shows — capture fidelity (no loss, correct order).
- **Non-intrusion test:** you forget it's running (zero perceptible hook latency).

## Phases & tasks

Check off tasks as completed. Each phase should leave the tree green.

### Accepted assumptions (spike skipped)

We're proceeding without a dedicated spike. These are taken as working assumptions and
validated empirically while building (cheap to correct via the `Raw` passthrough and the
daemon-as-buffer design):

- **Hooks are async / non-blocking** — Claude Code does not wait on the visualizer. The
  CLI's non-blocking guarantee + the ~100ms ceiling make a slow daemon a non-issue.
- **Payload sizes are bounded enough** that one `POST` per event over localhost is
  instant; if a pathological `tool_response` appears, the `io.LimitReader` cap catches it.
- **Hook events** are the documented set (SessionStart, SessionEnd, UserPromptSubmit,
  PreToolUse, PostToolUse, Notification, Stop, SubagentStop, PreCompact, …); the plugin
  manifest (Phase 4) is the place we'll discover the exact installed set.
- **Pre/Post pairing** uses a stable `tool_use_id` if present, else heuristic fallback
  (next Post of same tool in session) — see Phase 7.
- **Port strategy:** default `7842`, ephemeral fallback written to a port file.
